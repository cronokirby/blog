On The Malleability of ECDSA Signatures

  • #cryptography
  • #signature

The ECDSA signature scheme is quite ubiquitous, used everywhere from TLS to various cryptocurrencies like Bitcoin. Funnily enough, it turns out that it suffers from a few malleability issues, although I doubt these pose a serious issue in practice.

The way signature schemes work is relatively straightforward. Using my private-public key-pair (x,X)(x, X)(x,X), then I can sign a message mmm with private key, producing a signature σ=S(x,m)\sigma = S(x, m)σ=S(x,m). Anyone can verify this signature using my public key. They do this by running an algorithm V(X,m,σ)V(X, m, \sigma)V(X,m,σ), which checks if the signature is valid. If the signature scheme is secure, it shouldn't be possible for someone to forge a signature σ′\sigma'σ′ such that V(X,m,σ′)V(X, m, \sigma')V(X,m,σ′) returns true, unless they know the private key xxx.

That's the usual way these schemes are used. But, if you just have a message mmm, and a signature σ\sigmaσ, can you check that the signature was produced by the private key associated with XXX?

It turns out that for the ECDSA signature scheme (as used in Bitcoin, Ethereum, and many other places), the signature does not actually bind to a public key.

In particular, if you have a public key, a message, and a signature, which are valid together: (X,m,σ)(X, m, \sigma)(X,m,σ), you can construct two different kinds of triples that will also verify, if you swap out the public key for a different one.

The first method is to modify the message, giving you a valid:

(X′,m′,σ)(X', m', \sigma)(X′,m′,σ)

And the second method is to modify the signature, giving you a valid:

(X′,m,σ′)(X', m, \sigma')(X′,m,σ′)

In both cases, you don't need to know the private key, i.e. the discrete logarithm, associated with X′X'X′ to produce these signatures. This can violate many assumptions around how digital signatures are supposed to work, especially in the context of cryptocurrencies.

Recalling the ECDSA Scheme

Before we investigate how to forge these new signatures, we need to look at how the ECDSA signature scheme actually works. We'll be exploiting the inner mechanisms of the scheme, so understanding it is important.

We start with an Elliptic Curve group G\mathbb{G}G, having an associated generator GGG, and a scalar field Fq\mathbb{F}_qFq​.

We also have a hash function h:G→Fqh : \mathbb{G} \to \mathbb{F}_qh:G→Fq​, hashing points into scalars, as well as a hash function H:0,1∗→FqH : \\{0, 1\\}^* \to \mathbb{F}_qH:0,1∗→Fq​ hashing messages into scalars.

[!note] Note: In practice, the hash function hhh is actually just taking the xxx coordinate of an Elliptic Curve point, interpreted as an integer in Z\mathbb{Z}Z, and then reducing that value modulo qqq.

Key Generation

A private key is a random scalar x∈Fqx \in \mathbb{F}_qx∈Fq​, and the public key is the associated point X:=x⋅GX := x \cdot GX:=x⋅G.

Signing

To sign, you first choose a random nonce:

k←RFqk \xleftarrow{R} \mathbb{F}_qkR​Fq​

We then define a "commitment" to this nonce as:

R=1k⋅GR = \frac{1}{k} \cdot GR=k1​⋅G

At this point, if h(R)=0h(R) = 0h(R)=0, we restart the signature process with a different nonce. The point RRR makes up the first half of our signature.

The second half of our signature is a scalar, computed as:

s=k(H(m)+h(R)x)s = k(H(m) + h(R)x)s=k(H(m)+h(R)x)

with xxx being our private key.

Finally, our complete signature is the pair:

σ=(R,s)\sigma = (R, s)σ=(R,s)

Verification

To verify a signature σ=(R,s)\sigma = (R, s)σ=(R,s), we first make sure that h(R)≠0h(R) \neq 0h(R)=0 and s≠0s \neq 0s=0, and then we check that:

s⋅R=?H(m)⋅G+h(R)⋅Xs \cdot R \stackrel{?}{=} H(m) \cdot G + h(R) \cdot Xs⋅R=?H(m)⋅G+h(R)⋅X

This works out, because the 1k\frac{1}{k}k1​ inside RRR cancels with the kkk inside of sss, and because the public key XXX is equal to x⋅Gx \cdot Gx⋅G.

Sometimes you'll see 1k\frac{1}{k}k1​ used as the nonce instead, with the commitment being k⋅Gk \cdot Gk⋅G. This makes no difference whatsoever: it's just a matter of notation.

Secondly, schemes will often only transfer the xxx coordinate of RRR, because of the specific hash function used, and then rearrange the verification as:

h(R)=?h(1s(H(m)⋅G+h(R)⋅X))h(R) \stackrel{?}{=} h(\frac{1}{s}(H(m) \cdot G + h(R) \cdot X))h(R)=?h(s1​(H(m)⋅G+h(R)⋅X))

comparing only the xxx coordinates.

But this makes no difference as to how the scheme really works; it's just a bit of obfuscation, for our purposes.

Same Signature, Different Message

Now, let's tackle the first kind of malleability. Keeping the same signature, we'd like to find a new message m′m'm′ and public key X′X'X′ that will still verify with that signature.

Looking at our previous equation, we need to have:

s⋅R=H(m)⋅G+h(R)⋅Xs \cdot R = H(m) \cdot G + h(R) \cdot Xs⋅R=H(m)⋅G+h(R)⋅X

We can rearrange things so that XXX is on one side of the equation:

X=1h(R)(s⋅R−H(m)⋅G)X = \frac{1}{h(R)}(s \cdot R - H(m) \cdot G)X=h(R)1​(s⋅R−H(m)⋅G)

Now, since everything on that right side is public information, we can actually calculate XXX. What we've now done, essentially, is written the public key XXX as a function X(σ,m)X(\sigma, m)X(σ,m), of the signature and the message. I want to insist on the importance of this. With this function in place, we can choose any signature σ′\sigma'σ′ and message m′m'm′, and find a public key X′X'X′ which will validate this message and signature. At first glance, you might think that this would allow us to recover the public key that produced a given signature over some message. The problem is that we don't know which message the signature was made over.

In particular, if we plug in a different message m′m'm′ into this formula, we get a different public key X′X'X′:

X′=1h(R)(s⋅R−H(m′)⋅G)X' = \frac{1}{h(R)}(s \cdot R - H(m') \cdot G)X′=h(R)1​(s⋅R−H(m′)⋅G)

Note that since we don't know what the discrete logarithm of RRR is, we don't know what the discrete logarithm of X′X'X′ is either. Yet, we've managed to produce a signature under that public key, which is the issue here.

Same Message, Different Signature

We can also reuse this same formulation, but swap out the signature σ\sigmaσ instead, while keeping the message fixed:

X′=1h(R′)(s⋅R′−H(m)⋅G)X' = \frac{1}{h(R')}(s \cdot R' - H(m) \cdot G)X′=h(R′)1​(s⋅R′−H(m)⋅G)

Now, another way to do this is to create your own key-pair, and then sign the message yourself, producing a new triple (X′,m,σ′)(X', m, \sigma')(X′,m,σ′). The difference with our method here is that we don't know the discrete logarithm of X′X'X′. If you generated a new key-pair, you would naturally know the discrete logarithm.

Some Potential Issues

In all honesty, I don't know of any practical examples where this malleability has caused issues. In fact, I can only think of a few theoretical situations where this would matter.

Transactions in cryptocurrencies are usually accompanied with a signature, which attests that the author of the transaction. Naturally, to verify a transaction, you also need to know the author. One potential issue is if the author weren't included with the transaction, but rather derived using the transaction and the signature. This would have the problem of not being unique, as we've seen earlier.

In general, you can't uniquely identify transactions by their signatures, because of all the forms of malleability we've seen so far. That being said, I doubt that any system would actually end up trying to do this, but you never know what kind of shortcuts people will end up trying to take.

The second kind of malleability creates a different source of issues: we can produce a signature for a public key X′X'X′ without knowing the private key associated with X′X'X′. Now, in more intricate protocols, like threshold signatures, this can create a bit of an issue, but I'm not directly aware of how the kind of malleability we saw would lead to issues of this kind. But, once again, you never know what kind of ad-hoc protocols people will try and create on top of signatures.

How Schnorr Signatures Fix This (In Theory)

A popular alternative to ECDSA signatures are Schnorr Signatures, most commonly in the form of Ed25519 signatures. You can actually set up this signature scheme so that the signatures are tied to a specific public key, avoiding this kind of malleability.

I wrote a post on Schnorr Signatures previously, and I'd recommend taking a look at it for more information, but I'll recap the essence of the scheme here.

To sign a message mmm, you first generate a nonce:

k←RFqk \xleftarrow{R} \mathbb{F}_qkR​Fq​

And then a commitment to that nonce:

K=k⋅GK = k \cdot GK=k⋅G

Then, you generate a challenge:

e=H(K,m)e = H(K, m)e=H(K,m)

and then your response:

s=k+exs = k + exs=k+ex

The final signature is σ=(K,s)\sigma = (K, s)σ=(K,s).

This signature is verified by checking:

s⋅G=?K+H(K,m)⋅Xs \cdot G \stackrel{?}{=} K + H(K, m) \cdot Xs⋅G=?K+H(K,m)⋅X

(Omitting some ≠0\neq 0=0 checks which are also needed)

Now, what's interesting with Schnorr signatures is that we can actually add more elements to the challenge generation. Right now, we just hash the commitment to the nonce, and we also include the message. Including the message means that this signature is tied to this particular message, which is something we definitely want. If we include our public key:

e=H(K,X,m)e = H(K, X, m)e=H(K,X,m)

then this ties our signature to our public key as well, preventing the kinds of malleability we saw earlier.

We can go even further, and add in an additional piece of context:

e=H(K,X,ctx,m)e = H(K, X, \text{ctx}, m)e=H(K,X,ctx,m)

For example, we could add in a string identifying our kind of cryptocurrency, to prevent transactions being used in a different one. This might actually end up being relevant if different systems end up using the same transaction format, since you could end up sniffing a transaction on one system and submitting it to the other. By binding our signatures to a specific context, we disallow reusing signatures from one context in a different one.

[!note] Note: In practice, Schnorr Signatures are by no means perfect. The popular Ed25519, in particular, requires good attention to the details of encoding and verification, as elaborated in 1.

Summary

To summarize, given a message mmm, and an ECDSA signature σ\sigmaσ, it's possible to possible to find a public key X(m,σ)X(m, \sigma)X(m,σ) which allows the signature for that message to be verified. This means that two kinds of malleability exist. Given a triple (X,m,σ)(X, m, \sigma)(X,m,σ), you can change the public key and message, keeping the signature fixed:

(X′,m′,σ)(X', m', \sigma)(X′,m′,σ)

Or, you can change the public key and the signature, keeping the message fixed:

(X′,m,σ′)(X', m, \sigma')(X′,m,σ′)

In both cases, you do not need to know the discrete logarithm of X′X'X′, which can pose some issues, in theory.
