On Defining the One More Discrete Logarithm Problem

Here are three slightly different notions of the "One More Discrete Logarithm Problem" (OMDL), formulated as games.

[!note] Note: I'll be using the formalism of State-Separable proofs, as I wrote about recently.

First, the classic notion:

OMDL-Adaptiveb x_0,…,x_n+1←RZ/(q)X_0,…,X_n+1←xi⋅GPublic():‾ return (X_0,…,X_n+1)count←1DLog(i):‾ assert count++<n+1 return xiChallenge(x^_0,…,x^_n+1):‾ return b=0∧∀i. x^i=xi\boxed{ \begin{aligned} &\colorbox{#FBCFE8}{\large $\text{OMDL-Adaptive}_b$ }\cr \cr &x\_0, \ldots, x\_{n + 1} \xleftarrow{R} \mathbb{Z}/(q)\cr &X\_0, \ldots, X\_{n + 1} \gets x_i \cdot G\cr \cr &\underline{\texttt{Public()}:}\cr &\ \texttt{return } (X\_0, \ldots, X\_{n + 1})\cr \cr &\text{count} \gets 1\cr &\underline{\texttt{DLog(}i\texttt{)}:}\cr &\ \texttt{assert } \text{count++} < n + 1\cr &\ \texttt{return } x_i\cr \cr &\underline{\texttt{Challenge(}\hat{x}\_0, \ldots, \hat{x}\_{n + 1}\texttt{)}:}\cr &\ \texttt{return } b = 0 \land \forall i.\ \hat{x}_i = x_i\cr \end{aligned} }​OMDL-Adaptiveb​ ​x_0,…,x_n+1R​Z/(q)X_0,…,X_n+1←xi​⋅GPublic():​ return (X_0,…,X_n+1)count←1DLog(i):​ assert count++<n+1 return xi​Challenge(x^_0,…,x^_n+1):​ return b=0∧∀i. x^i​=xi​​​

In this game, the adversary is given n+1n + 1n+1 group elements, and can query for nnn of the discrete logarithms, adaptively.

Another variant allows queries for elements of the adversary's choice, but not adaptively. This game is identical, with the DLog\texttt{DLog}DLog being the only change:

OMDL-Dynamicb …count←1DLog(i^):‾ assert count++=1 return (xi ∣ i≠i^)\boxed{ \begin{aligned} &\colorbox{#FBCFE8}{\large $\text{OMDL-Dynamic}_b$ }\cr &\ldots \cr &\text{count} \gets 1\cr &\underline{\texttt{DLog(}\hat{i}\texttt{)}:}\cr &\ \texttt{assert } \text{count++} = 1\cr &\ \texttt{return } (x_i\ |\ i \neq \hat{i})\cr \end{aligned} }​OMDL-Dynamicb​ ​…count←1DLog(i^):​ assert count++=1 return (xi​ ∣ i=i^)​​

Finally, you can also make it so that the adversary doesn't get to choose which group elements they get the logarithm for:

OMDL-Staticb …DLog():‾ return (xi ∣ i≠n+1)\boxed{ \begin{aligned} &\colorbox{#FBCFE8}{\large $\text{OMDL-Static}_b$ }\cr &\ldots \cr &\underline{\texttt{DLog(}\texttt{)}:}\cr &\ \texttt{return } (x_i\ |\ i \neq n + 1)\cr \end{aligned} }​OMDL-Staticb​ ​…DLog():​ return (xi​ ∣ i=n+1)​​

Now, because each of these gives progressively weaker capabilities, we have:

OMDL-Staticb≤OMDL-Dynamicb≤OMDL-Adapativeb\text{OMDL-Static}_b \leq \text{OMDL-Dynamic}_b \leq \text{OMDL-Adapative}_bOMDL-Staticb​≤OMDL-Dynamicb​≤OMDL-Adapativeb​

This can be proved via straight-line reductions.

The other direction is harder.

OMDL-Dynamicb≤OMDL-Staticb\text{OMDL-Dynamic}_b \leq \text{OMDL-Static}_bOMDL-Dynamicb​≤OMDL-Staticb​

The idea behind this reduction is that if the adversary A\mathcal{A}A for OMDL-Dynamic\text{OMDL-Dynamic}OMDL-Dynamic happens to ask for i^=n+1\hat{i} = n + 1i^=n+1, you can satisfy their query immediately. At first, this seems like it would reduce your advantage by a factor of n+1n + 1n+1, since you need to get lucky.

The trick is that you can reset the adversary, and try again. The adversary may not ever pick n+1n + 1n+1 in their strategy, so instead you need to rearrange the group elements yourself, picking where you place the "hole" randomly, so that there's always a 1/(n+1)1 / (n + 1)1/(n+1) chance of the adversary picking it.

Proof:

More formally, given this adversary A\mathcal{A}A for OMDL-Dynamic\text{OMDL-Dynamic}OMDL-Dynamic, we construct an adversary B\mathcal{B}B for OMDL-Static\text{OMDL-Static}OMDL-Static, as follows:

  1. Query Public\texttt{Public}Public and DLog\texttt{DLog}DLog to learn X0,…,Xn+1X_0, \ldots, X_{n + 1}X0​,…,Xn+1​ and associated x0,…,xnx_0, \ldots, x_nx0​,…,xn​. Let xn+1=⊥x_{n + 1} = \botxn+1​=⊥.
  2. Retry the following steps until they succeed (resetting any state modifications):
  3. Choose random r_0,…,r_n+1←RZ/(q)r\_0, \ldots, r\_{n + 1} \xleftarrow{R} \mathbb{Z}/(q)r_0,…,r_n+1R​Z/(q), and set xj←rj⋅xjx_j \gets r_j \cdot x_jxj​←rj​⋅xj​, as well as Xj←rj⋅XjX_j \gets r_j \cdot X_jXj​←rj​⋅Xj​.

The expected number of retries is n+1n + 1n+1. This is because the behavior of A\mathcal{A}A is completely independent of jjj, particularly because we randomize the group elements at each iteration, so that they're indistinguishable from freshly sampled group elements. Thus, at each try, there's at most an n/(n+1)n / (n + 1)n/(n+1) chance that we have to abort when running A\mathcal{A}A.

□\square□

OMDL-Adaptiveb≤OMDL-Staticb\text{OMDL-Adaptive}_b \leq \text{OMDL-Static}_bOMDL-Adaptiveb​≤OMDL-Staticb​

It's actually not any easier to prove OMDL-Adaptive≤OMDL-Dynamic\text{OMDL-Adaptive} \leq \text{OMDL-Dynamic}OMDL-Adaptive≤OMDL-Dynamic, so we'll prove this one instead.

The basic idea of the reduction is the same, it's just that the analysis is more complicated. We re-randomize and shuffle the group elements as in the previous reduction, and we abort the adversary A\mathcal{A}A if it ever queries the group element we have no scalar for. Analyzing this is trickier because of the adaptivity.

The idea is that given our choices of group elements X_0,…,X_n+1X\_0, \ldots, X\_{n + 1}X_0,…,X_n+1, we can look at the sequence of queries made by A\mathcal{A}A, if they get the answer each time. This defines a sequence of random variables i0,…,ini_0, \ldots, i_ni0​,…,in​, which depends solely on XiX_iXi​. We can also define a related random variable, i^\hat{i}i^, represented the index which isn't queried. This is a function of i0,…,ini_0, \ldots, i_ni0​,…,in​, and so depends solely on our choice of XiX_iXi​.

In particular, i^\hat{i}i^ is independent from the random jjj we choose at each iteration, because of the re-randomization, and so the analysis from the previous reduction applies exactly.

□\square□
